# Dirty State and Versioned State

## Architecture decision

`GeneratedMetadata` is a **Versioned** DataObject saved via sideways XHR (independent from the page edit form). It participates in the Draft/Live cycle but is published alongside the parent page, not independently. This matches the editor mental model that "publish page = publish everything on the page", while `ReviewedAt` prevents unverified AI content from going live.

## How it works

- **Metadata writes go to Draft.** The CMS modal submits directly to the `GeneratedMetadata` record via XHR. The `write()` call saves to the Draft stage. The page edit form is not involved.
- **No dirty state interaction.** Editing or regenerating metadata does not mark the page form as having unsaved changes.
- **Metadata is published when the page is published.** A JS Entwine hook fires after the page publish action and sends an XHR to `AiMetadataController::publishMetadata()`. This mirrors the Elemental pattern.
- **ReviewedAt publish gate.** The publish endpoint only publishes metadata if `ReviewedAt` is set **and** `ReviewedAt >= GeneratedAt` on the Draft record (i.e. `isReviewed()` returns true). Unreviewed metadata (e.g. generated by the background job but not yet human-reviewed) stays in Draft even when the page is published. This prevents unverified AI content from going live.
- **Background job writes to Draft** with `ReviewedAt = null`. The metadata sits in Draft awaiting human review.
- **Page unpublish/archive cascades.** When a page is unpublished or archived, the hook unpublishes the metadata too.

## Publish scenarios

### Scenario 1: Editor generates + reviews + publishes

1. Editor opens modal, generates metadata, reviews, submits → Draft with `ReviewedAt` set
2. Editor publishes page → hook publishes metadata → metadata is Live

### Scenario 2: Background job generates, editor publishes for unrelated reason

1. Background job generates metadata → Draft with `ReviewedAt = null`
2. Editor publishes page → hook sees `ReviewedAt` is null → skips publish → old Live metadata (if any) stays, unreviewed Draft stays
3. Editor later reviews metadata in modal → `ReviewedAt` set on Draft
4. Editor publishes page → hook publishes metadata → new metadata is Live

### Scenario 3: Content changes after reviewed metadata is live

1. Reviewed metadata is Live. Content changes, background job re-generates → new Draft with `ReviewedAt = null`
2. Editor publishes page → hook sees `ReviewedAt` is null → skips → old reviewed Live metadata stays
3. Editor reviews new metadata → `ReviewedAt` set
4. Next publish → new metadata goes Live

## Review gating

- The `ReviewedAt` field on `GeneratedMetadata` tracks when an editor last reviewed and submitted metadata
- `ReviewedAt` is set when the editor submits via the modal
- `ReviewedAt` is reset to null when metadata is regenerated (by background job or manual regeneration)
- `ReviewedAt` serves as the **publish-readiness gate** - only reviewed metadata (where `ReviewedAt >= GeneratedAt`, checked via `isReviewed()`) participates in the page publish cascade
- The CMS report uses `ReviewedAt` to identify pages with unreviewed metadata
